# Архитектура движка JS (V8): Память и Стек

Движок JavaScript (например, V8 в Chrome и Node.js) состоит из двух главных компонентов, которые работают в тандеме: **Memory Heap** (Куча) и **Call Stack** (Стек вызовов).

## Memory Heap (Куча памяти)

**Что это:** Это большая, неструктурированная область памяти. Можно представить ее как огромный склад, где вещи лежат хаотично, но у каждой есть свой уникальный адрес.

**Что здесь хранится:** Здесь выделяется память под **ссылочные типы данных** (Reference Types):

- Объекты (`{}`)
- Массивы (`[]`)
- Функции (`function() {}`)

*Примечание:* Примитивы (числа, строки, булевы значения) обычно хранятся прямо в Call Stack (в контексте выполнения), а в Кучу отправляются только тяжелые и динамические структуры.

### Как работает Garbage Collector (Сборщик мусора)

JS имеет автоматическое управление памятью. Нам не нужно вручную очищать память, как в C++. Этим занимается Garbage Collector (GC) с помощью алгоритма **Mark-and-Sweep (Пометь и выброси)**.

**Алгоритм Mark-and-Sweep по шагам:**

1. **Roots (Корни):** GC начинает работу с "корней". В браузере главным корнем является глобальный объект `window` (в Node.js — `global`).
2. **Mark (Пометка):** GC идет от корней и помечает все объекты, до которых может "дотянуться" по ссылкам. Если объект A ссылается на объект B, то объект B тоже помечается как "достижимый".
3. **Sweep (Очистка):** GC проходит по всей памяти. Все объекты, которые **не получили пометку** (то есть до них нельзя добраться из глобального объекта, они стали "островами"), считаются мусором. Память из-под них освобождается.

## Call Stack (Стек вызовов)

**Что это:** Это структура данных, которая работает по принципу **LIFO (Last In, First Out — Последним пришел, первым ушел)**. Стек отслеживает, в каком месте кода мы сейчас находимся и какие функции выполняются.

**Как JS выполняет код (Синхронность):**

JavaScript — **однопоточный** язык. У него только один Call Stack. Это значит, что он может выполнять только *одну задачу в единицу времени*. Код выполняется кадр за кадром (Frame by Frame).

**Что такое Execution Context (Контекст выполнения):**

Каждый раз, когда вызывается функция, для нее создается свой *Контекст выполнения*, который помещается на самый верх стека.

1. При запуске скрипта на дно стека кладется **Global Execution Context**.
2. Вызываем функцию `A()` — она кладется поверх глобального.
3. Внутри `A()` вызываем `B()` — она кладется поверх `A()`.
4. Когда `B()` отрабатывает до конца (возвращает `return` или заканчивается код), она **снимается** со стека (pop).
5. JS продолжает выполнение `A()`.

### Что такое переполнение стека (Stack Overflow)

Так как размер стека ограничен (обычно около 10 000 вызовов в V8), его можно переполнить. Это происходит, если мы добавляем функции в стек быстрее, чем они успевают оттуда удаляться.

**Классический пример:** Бесконечная рекурсия без базового случая (условия выхода).

```jsx
function inception() {
  inception(); // Функция вызывает саму себя бесконечно
}

inception(); // Uncaught RangeError: Maximum call stack size exceeded
```
