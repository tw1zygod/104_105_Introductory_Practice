# Теория к первому занятию

## Что происходит, когда вы открываете google.com?

### Этап 1: сетевое взаимодействие

1. **IP-адресация и DNS-резолвинг:**

Вычислительные машины в сети взаимодействуют друг с другом исключительно с помощью числовых идентификаторов, называемых IP-адресами. Поскольку человеку сложно запоминать массивы чисел, была разработана система доменных имен (DNS), которая выполняет функцию глобальной «телефонной книги» интернета. Когда вы вводите текстовый адрес (например, `google.com`), браузер сначала обращается к DNS-серверу с запросом на преобразование этого имени в конкретный IP-адрес. Получив в ответ числовое значение, браузер определяет точный сетевой узел, к которому необходимо подключиться для получения сайта.

2. **Клиент-серверная архитектура и HTTP-запрос:**

Весь современный веб работает на основе архитектуры «Клиент — Сервер». Клиент (в данном случае — браузер) выступает инициатором: он формирует запрос на получение конкретной страницы или файла. Обмен данными между браузером и сервером строго регламентирован протоколом HTTP (или его защищенной версией HTTPS). Любая транзакция в этом протоколе состоит из неразрывной пары: отправки запроса (HTTP Request) клиентом и получения ответа (HTTP Response) от сервера.

3. **Обработка сервером:**

Сервер — это мощная вычислительная машина, работающая непрерывно, чья главная функция заключается в обработке входящих запросов от клиентов и выдаче соответствующих файлов (кода, стилей, изображений). Сервер Google принимает поисковой запрос, вычисляет результаты и отправляет их обратно **в виде текстового кода**.

### Этап 2: интерпретация и отрисовка

Получив от сервера текстовый код (HTML, CSS), браузер начинает его интерпретацию. За этот процесс отвечает специальный программный модуль — движок рендеринга (Rendering Engine). Далее браузер преобразует текстовый ответ сервера в графический интерфейс — это называется рендерингом, который включает несколько этапов:

1. **Парсинг HTML и построение DOM-дерева:**

Браузер осуществляет последовательный динамический разбор (парсинг) полученного HTML-документа сверху вниз. Текстовые теги преобразуются в объектную модель документа — DOM (Document Object Model). DOM представляет собой иерархическое дерево узлов, которое формирует каркас страницы. Если в процессе разбора браузер встречает тег `<script>`, он может приостановить построение DOM, чтобы выполнить код JavaScript, так как этот код способен изменить структуру страницы.

2. **Обработка стилей и построение CSSOM-дерева:**

Параллельно с HTML браузер анализирует таблицы стилей (CSS). На их основе строится дерево CSSOM (CSS Object Model), которое хранит правила визуального оформления (цвета, отступы, шрифты) для каждого узла.

3. **Формирование дерева рендеринга (Render Tree):**

Браузер объединяет данные из DOM и CSSOM в единую структуру. Главная особенность дерева рендеринга заключается в том, что в него включаются только видимые элементы; скрытые и служебные теги из него исключаются.

4. **Вычисление геометрии (Layout / Reflow):**

Опираясь на дерево рендеринга, браузер математически рассчитывает точные координаты, ширину и высоту каждого структурного блока, учитывая текущие физические размеры окна просмотра (viewport).

5. **Отрисовка пикселей (Painting):**

На финальном этапе браузер переводит рассчитанные геометрические блоки в фактические пиксели на матрице монитора, осуществляя послойную отрисовку текста, границ, теней и изображений.

## Что такое HTTP(s)?

**HTTP (HyperText Transfer Protocol)** — это специальный протокол (язык общения), который строго регламентирует обмен данными между клиентом (например, веб-браузером) и сервером. Любое общение (транзакция) в рамках этого протокола всегда состоит из неразрывной пары действий:

- HTTP Request (Запрос) — формируется и отправляется клиентом, чтобы получить страницу, файл или данные.

- HTTP Response (Ответ) — возвращается сервером после обработки запроса.

Также существует защищенная (безопасная) версия этого протокола, которая называется HTTPS (где «S» означает Secure). В ней все данные передаются в зашифрованном виде.

### Из чего состоит HTTP-сообщение

**Пример 1:** HTTP-запрос (Request) от клиента

Представим, что пользователь ввел адрес сайта и браузер запрашивает главную страницу. Для этого используется метод GET (запрос на получение данных).

```http
GET / HTTP/1.1
Host: example.com
User-Agent: curl/8.20.0
Accept: */*
```

Из чего он состоит:

- **Стартовая строка (Start Line):** `GET / HTTP/1.1`. Она указывает метод (GET — запрос на получение данных ), путь (/ — означает обращение к главной странице ) и используемую версию протокола.

- **Заголовки (Headers):** Это блок служебной информации.

    - `Host: example.com`: Обязательный заголовок, который указывает доменное имя целевого сервера.
    - `User-Agent: curl/8.20.0`: Строка, в которой клиент рассказывает о себе. В данном примере мы видим, что запрос был отправлен не из обычного браузера, а через консольную программу curl версии 8.20.0.
    - `Accept: */*`: Служебный заголовок, сообщающий серверу, что клиент готов принять ответ в абсолютно любом формате.

- **Пустая строка:** Обязательный разделитель, показывающий, что блок служебной информации завершен.

- **Тело сообщения (Body):** Здесь оно отсутствует, так как в GET-запросах на получение страницы полезная нагрузка в теле обычно не передается.


**Пример 2:** HTTP-ответ (Response) от сервера

Сервер получил наш запрос, нашел нужную страницу и возвращает ее обратно браузеру.

```http
HTTP/1.1 200 OK
Date: Sat, 13 Jun 2026 14:23:27 GMT
Content-Type: text/html
Transfer-Encoding: chunked
Connection: keep-alive
Server: cloudflare
last-modified: Tue, 09 Jun 2026 21:01:00 GMT
allow: GET, HEAD
Accept-Ranges: bytes
Age: 5051
cf-cache-status: HIT
CF-RAY: a0b1c0d29f6066eb-AMS

<!doctype html><html lang="en">...
```

Из чего он состоит:

- **Стартовая строка (Status Line):** `HTTP/1.1 200 OK`. Сервер сообщает версию протокола и статус выполнения — 200 OK, что означает успешную обработку нашего запроса.

- **Заголовки:** Обширный блок метаданных
    - `Content-Type: text/html`: Один из самых важных параметров, сообщающий, что в теле ответа будет находиться именно HTML-код сайта.
    - `Server: cloudflare`: Сообщает информацию об инфраструктуре сервера.
    - `Date, last-modified, cf-cache-status: HIT и другие`: Различные служебные параметры, передающие контекст о времени создания документа, правилах соединения и статусе кэширования.

- **Пустая строка**: Обязательный разделитель.

- **Тело сообщения:** Полезная нагрузка ответа. В нашем случае это весь блок, начиная с `<!doctype html><html lang="en">...` — сам HTML-код, который клиент должен будет прочитать и отрисовать.

### Типы запросов (HTTP Methods)

Чтобы сервер понимал, какое именно действие клиент хочет совершить с данными, стартовая строка запроса всегда начинается с метода (глагола).

Основные методы (CRUD-операции):

- **GET:** Запрос на получение данных. (Например: «Дай мне главную страницу» или «Покажи картинку»). В GET-запросах обычно нет тела (Body), вся информация передается в URL.

- **POST:** Запрос на отправку новых данных на сервер. (Например: отправка формы регистрации или загрузка фотографии). Данные надежно спрятаны в теле (Body) запроса.

- **PUT / PATCH:** Запросы на обновление или изменение уже существующих данных на сервере. (Например: изменение имени в профиле пользователя).

- **DELETE:** Запрос на удаление данных с сервера.

### Заголовки

**Заголовки** — это служебные параметры, записанные в формате Имя-заголовка: Значение. Они передают важный контекст как от клиента серверу, так и наоборот.

Примеры популярных заголовков:

- **Host:** Указывает доменное имя сервера (например, google.com), к которому идет обращение. Обязателен в HTTP/1.1.

- **User-Agent:** Строка, в которой браузер рассказывает о себе (какая операционная система, версия Chrome/Safari и т.д.). Так сервер понимает, кто к нему пришел (мобилка или ПК).

- **Content-Type:** Говорит, в каком формате передается тело сообщения (например, text/html для кода сайта или application/json для данных API).

- **Authorization:** Передает серверу ключи доступа или токены, чтобы доказать, что у пользователя есть права на просмотр закрытой страницы.

### Коды состояния

Когда сервер формирует ответ, в его стартовой строке всегда присутствует трехзначное число — код состояния (например, 200 или 404). Именно по этому коду браузер (и разработчик) мгновенно понимает результат запроса, даже не заглядывая в само тело ответа.

Все коды глобально разделены на 5 классов (или сотен), где первая цифра определяет общую категорию ответа:

- **1xx (Информационные):** «Я получил твой запрос, процесс идет, подожди». Служебные коды, с которыми в повседневной работе почти не сталкивается обычный пользователь.

- **2xx (Успех):** «Всё отлично, я всё сделал». Означает, что сервер успешно получил, понял и обработал запрос клиента.

- **3xx (Перенаправление):** «То, что ты ищешь, теперь находится в другом месте». Сервер сообщает, что для успешного выполнения запроса клиенту нужно сделать дополнительное действие (например, перейти по другому URL).

- **4xx (Ошибка клиента):** «Ты где-то ошибся». Самая частая ошибка. Этот класс кодов говорит о том, что сервер работает нормально, но сам запрос составлен неверно: в URL опечатка, не хватает прав доступа или отправлены неправильные данные.

- **5xx (Ошибка сервера):** «Ты всё сделал правильно, но я сломался». Это означает, что клиент отправил хороший запрос, но сервер не смог его обработать из-за внутренней ошибки (например, упала база данных или сервер физически перегружен).

Все типы кодов ответов можно посмотреть на сайте с котиками — [здесь :3](https://http.cat/)

### Версии протокола: Почему мы изучаем HTTP/1

Протокол HTTP развивался годами. Сегодня существуют современные версии: HTTP/2 и HTTP/3. Они работают быстрее, умеют отправлять несколько файлов одновременно (мультиплексирование) и работают на бинарном уровне, а не на текстовом.
Однако для понимания основ веба мы всегда обращаемся к классической версии — HTTP/1.1. Ее главная особенность: она полностью текстовая. Это значит, что запросы и ответы в HTTP/1 можно легко прочитать глазами, как обычный текстовый документ.

- **HTTP/1.1 (Классика и база)**: Эта версия нужна для понимания основ работы веба. Ее главное отличие заключается в том, что она полностью текстовая, поэтому разработчики могут легко читать запросы и ответы глазами, как обычный документ. Однако она плохо справляется с загрузкой современных сайтов, где нужно скачивать десятки картинок и скриптов одновременно.

- **HTTP/2 (Скорость и мультиплексирование)**: Понадобилась, когда сайты стали сложными и «тяжелыми». Чтобы работать быстрее, эта версия перешла на бинарный уровень и научилась отправлять несколько файлов одновременно по одному соединению (это называется мультиплексированием).

- **HTTP/3 (Надежность в мобильную эру)**: Создана для работы в условиях нестабильного интернета (например, на смартфонах). Она также работает быстрее и поддерживает мультиплексирование, но при этом использует другой транспортный фундамент (переход с TCP на UDP/QUIC). Это позволяет не терять скорость загрузки при потере пакетов данных или при переключении телефона с домашнего Wi-Fi на мобильную сеть LTE.

## Что такое HTML?

**HTML (HyperText Markup Language)** — это язык гипертекстовой разметки. В настоящих языках программирования (как JavaScript, Python или C++) есть математическая логика: переменные, условия («если произошло X, сделай Y») и циклы. В HTML ничего этого нет, он работает исключительно как набор указаний (ярлыков) для браузера о том, как структурировать текст.

**HTML** — это декларативный язык. Мы не объясняем браузеру, как нарисовать кнопку. Мы просто декларируем факт: «Вот здесь должна быть кнопка, а здесь — таблица». Всю математику по ее отрисовке браузер берет на себя.

### Из чего состоит HTML-элемент?

Вся разметка строится на тегах — специальных командах, заключенных в угловые скобки `< >`. Давайте возьмем сложный элемент под микроскоп и разберем его по атомам. Представим классическую ссылку: `<a href="https://ya.ru" target="_blank">Перейти в Яндекс</a>`

Этот элемент состоит из 4 частей:

- **Открывающий тег (`<a>`):** Указывает браузеру, где начинается элемент. Сам по себе тег `<a>` (anchor) говорит, что это ссылка.

- **Атрибуты (href="..." и target="..."):** Это дополнительные настройки тега, которые всегда пишутся внутри открывающего тега через пробел. Атрибут состоит из имени и значения, заключенного в кавычки (имя="значение"). В некоторых случаях значения может отсутствовать. В нашем примере атрибут `href` указывает, куда ведет ссылка, а `target="_blank"` приказывает браузеру открыть ее в новой вкладке.

- **Содержимое / Контент (Перейти в Яндекс):** Это видимый текст, по которому кликнет пользователь.

- **Закрывающий тег (`</a>`):** Обязательно содержит косую черту (/) и говорит браузеру: «Здесь ссылка закончилась, дальше идет обычный текст».

### Семантика

**Семантика** — это подход, при котором мы используем HTML-теги строго по их прямому смысловому назначению.

Представьте, что вы читаете газету. Вы легко понимаете, где заголовок, где главная статья, а где мелкий шрифт с адресом редакции, потому что визуально они отличаются. Но компьютер «слеп». Для него сайт — это просто сплошная стена текста. Семантический тег работает как ярлык, который четко объясняет и браузеру, и разработчику, и поисковому роботу: «Какая именно информация здесь находится?».

**Зачем использовать семантику?**

**Причина 1:** Читаемость кода и поддержка проекта.

Когда проект разрастается до тысяч строк кода, в нем легко заблудиться. Разработчику гораздо проще визуально сканировать код, разделенный на понятные `<header>`, `<main>` и `<section>`, чем пытаться распутать бесконечную «лапшу» из одинаковых `<div>`.

**Причина 2:** Доступность (Accessibility / a11y) и Инклюзивность.

Интернет должен быть удобен для всех, включая слепых и слабовидящих пользователей. Они взаимодействуют с сайтами через скринридеры — специальные программы экранного доступа, которые читают исходный код вслух.

Проблема в том, что скринридер не видит ваших красивых дизайнов. Если шапка сайта сделана через `<div>`, диктор не поймет, что это шапка, и просто монотонно прочитает список ссылок. Но если вы использовали `<header>`, диктор уважительно сообщит: «Здесь находится шапка сайта». Более того, благодаря тегу `<main>`, слепой пользователь может нажать специальную горячую клавишу и мгновенно перепрыгнуть через длинное скучное меню прямо к чтению главной статьи. Вы делаете интернет доступнее, просто используя правильные теги. Более того, у клиента вообще может отсутствовать дисплей.

**SEO (Search Engine Optimization)** — это поисковая оптимизация. Это набор правил, которые помогают вашему сайту появляться на первых строчках в Google или Яндексе.

**Поисковые роботы (краулеры)** — это программы, которые постоянно сканируют интернет. Робот не видит красивых картинок или анимаций, он читает только голый HTML-код. Задача разработчика — написать код так, чтобы роботу было легко его понять.

**3 золотых правила базового SEO в HTML:**

1. Метатеги в `<head>` (Лицо сайта для поисковика)

	- `<title>Название страницы</title>`: Это самый важный тег для SEO. Именно этот текст отображается крупным синим шрифтом в выдаче Google и на вкладке браузера.
	- `<meta name="description" content="Краткое описание сайта...">`: Этот текст выводится серым шрифтом под заголовком в поиске. Он должен коротко и заманчиво описывать суть страницы, чтобы пользователь захотел кликнуть.

2. Строгая иерархия заголовков (h1-h6)

Поисковый робот читает заголовки, чтобы понять структуру текста, как оглавление в книге.

- **Правило №1**: На каждой странице должен быть строго один (и только один!) тег `<h1>`. Это название «книги».
- **Правило №2**: Нельзя нарушать порядок. За `<h1>` должны идти разделы `<h2>`, внутри них — подразделы `<h3>` и так далее. Нельзя перепрыгивать через уровни (например, ставить `<h4>` сразу после `<h2>` просто потому, что вам больше нравится его размер шрифта).

3. Атрибут alt для изображений

Поисковые роботы не умеют «видеть» картинки, они читают их текстовое описание в атрибуте alt. Если у вас интернет-магазин кроссовок, фотография должна выглядеть так: `<img src="sneakers.jpg" alt="Белые беговые кроссовки Nike Air">`. Это не только помогает слабовидящим людям, но и позволяет картинкам вашего сайта попадать в поиск по изображениям (Google Картинки), принося дополнительный трафик.

**Примеры семантических тегов:**

- `<header>` (Шапка): Вводная часть страницы или отдельного смыслового блока. Обычно здесь живет логотип компании и блок с контактами.

- `<nav>` (Навигация): Блок с главными ссылками меню. Важное правило: не каждую группу ссылок на сайте нужно оборачивать в `<nav>`. Этот тег приберегают только для самой главной навигации (например, меню в шапке).

- `<main>` (Главный контент): Сердце страницы. Здесь находится уникальное содержимое, ради которого пользователь вообще открыл эту ссылку. Железное правило: на одной странице может быть строго один `<main>`.

- `<section>` (Тематический раздел): Смысловой блок внутри страницы. Например, длинный лендинг делится на секции: «О нас», «Наши услуги», «Отзывы». В отличие от `<article>`, секция обычно теряет контекст, если ее вырвать из страницы. Правило хорошего тона: у каждой секции обязательно должен быть свой заголовок (от `<h2>` до `<h6>`).

- `<footer>` (Подвал): Заключительная часть сайта. Здесь располагаются копирайты, юридическая информация, ссылки на соцсети и дублирующее меню.

Остальные теги вы можете найти [тут](https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Elements).

### Теги `<div>` и `<span>`

Существуют два классических «безликих» тега-контейнера, которые не несут в себе вообще никакого смысла: `<div>` и `<span>`. Они отличаются только поведением на странице:

- `<div>` (Division) — это блочный контейнер (коробка). Он всегда начинается с новой строки и стремится занять всю доступную ширину экрана.

- `<span>` — это строчный контейнер. Он не переносит текст на новую строку и обхватывает ровно столько контента, сколько в нем лежит (идеально, чтобы покрасить одно слово внутри длинного абзаца).

Долгое время весь интернет строился исключительно на этих контейнерах. В начале 2000-х сайты стали сложнее: появились боковые колонки, карточки товаров, сложные меню. Чтобы управлять всем этим в CSS (задавать цвета, выстраивать сетки), разработчикам нужно было оборачивать каждый логический кусок сайта в контейнер.

Поскольку семантических тегов (вроде `<header>` или `<nav>`) в старых версиях HTML еще не существовало, разработчики для всего использовали `<div>`.

В результате код сложных страниц стал выглядеть как нечитаемая каша, состоящая из сотен вложенных друг в друга коробок:

```html
<div id="header">
  <div class="logo">...</div>
  <div id="navigation">
    <div class="menu-item">...</div>
  </div>
</div>
```

Это явление получило в индустрии название **«Div Soup»** (Суп из дивов).

Визуально для пользователя сайт выглядел нормально. Но для разработчиков, которые пытались в этом коде разобраться, а главное — для поисковых роботов (Google) и программ для слепых людей, сайт выглядел как бесконечная гора одинаковых пустых картонных коробок, на которых просто написаны разные слова. Роботы не понимали, где здесь меню, а где главная статья.

Именно для того, чтобы навести порядок в этом хаосе, была придумана семантика в стандарте HTML5.

### Пример HTML страницы

```html
<!DOCTYPE html>
<html lang="ru">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Моя первая страница</title>
    <meta name="description" content="Пример правильной структуры современного HTML5 сайта.">
</head>
<body>
    <header>
        <h1>Мой Блог</h1>
        <nav>
            <ul>
                <li><a href="#">Главная</a></li>
                <li><a href="#">Статьи</a></li>
                <li><a href="#">Контакты</a></li>
            </ul>
        </nav>
    </header>

    <main>
        <section>
            <h2>Добро пожаловать!</h2>
            <p>Это пример того, как правильно использовать семантические теги.</p>
        </section>

        <article>
            <h2>Как выучить HTML за неделю</h2>
            <p>Для начала нужно понять разницу между обычными дивами и умными тегами...</p>
            <img src="study.jpg" alt="Студент изучает программирование за ноутбуком">
        </article>
    </main>

    <aside>
        <h3>Популярные теги</h3>
        <ul>
            <li><a href="#">Что такое Flexbox?</a></li>
            <li><a href="#">Основы JavaScript</a></li>
        </ul>
    </aside>

    <footer>
        <p>&copy; 2026 год</p>
    </footer>
</body>
</html>
```
