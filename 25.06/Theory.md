# Теория к девятому занятию

## События в JS

### Парадигма событийно-ориентированного программирования

**1. Проблема синхронного выполнения**
До девятого дня весь наш JavaScript-код выполнялся линейно и синхронно: как только страница загружалась, скрипт отрабатывал сверху вниз и завершал свою работу. Но современные веб-приложения так не работают. Они должны «ждать» действий пользователя (нажатия кнопки, ввода текста, прокрутки экрана) и реагировать на них в непредсказуемые моменты времени.

**2. Архитектура событий (Events)**
Для реализации такого поведения браузеры используют парадигму **событийно-ориентированного программирования**.
Событие (Event) — это системный сигнал от браузера о том, что в DOM-дереве что-то произошло. Движок JavaScript умеет подписываться на эти сигналы и запускать определенные участки кода только тогда, когда сигнал срабатывает.

### Регистрация обработчиков: Метод `addEventListener`

Чтобы научить HTML-узел реагировать на события, мы должны «повесить» на него слушателя (Event Listener). Для этого в DOM API существует фундаментальный метод `addEventListener()`.

**Синтаксис и аргументы:**
Метод принимает два обязательных параметра:

1. **Имя события (String):** Строка, описывающая тип сигнала (например, `'click'`, `'scroll'`, `'keydown'`). Пишется строго строчными буквами без префикса "on".
2. **Колбэк-функция (Callback):** Функция, которая будет вызвана браузером *в будущем*, строго в момент наступления события.

```javascript
// 1. Селекция: находим кнопку в DOM
const burgerButton = document.querySelector('.burger-menu');

// 2. Регистрация слушателя
burgerButton.addEventListener('click', function() {
    // Этот код НЕ выполнится при загрузке страницы!
    // Он будет срабатывать каждый раз, когда пользователь кликает по кнопке.
    console.log("Пользователь открыл меню!");
});
```

### Фундаментальные типы событий

Спецификация браузеров описывает сотни различных событий, но для проектирования базового интерактива (бургер-меню, модальные окна, фильтры) достаточно знать два основных:

* **Событие `click` (Клик мыши / Тап по экрану):**
Срабатывает, когда пользователь наводит курсор на элемент и совершает полное нажатие левой кнопкой мыши (или касается элемента на мобильном устройстве). Это главное событие для кнопок, ссылок и переключателей табов.
* **Событие `input` (Ввод данных):**
Применяется исключительно к интерактивным элементам форм (`<input>`, `<textarea>`). Срабатывает **моментально** при любом изменении значения в поле (при нажатии каждой новой клавиши, вставке текста из буфера обмена или удалении символов). Идеально для создания «живого поиска» или валидации пароля на лету.

```javascript
const searchField = document.querySelector('.search-input');

searchField.addEventListener('input', () => {
    console.log("Пользователь сейчас печатает...");
});
```

### Инкапсуляция метаданных: Объект `Event`

Когда происходит событие, браузер не просто вызывает вашу колбэк-функцию. Он собирает все физические параметры этого события (координаты мыши, какая кнопка была нажата, какой именно элемент спровоцировал событие) и упаковывает их в специальный объект — **Объект события (Event Object)**.

Браузер автоматически передает этот объект в качестве **первого аргумента** в вашу колбэк-функцию. По инженерной конвенции этот параметр принято называть `event` или сокращенно `e`.

**Критически важные свойства и методы объекта `Event`:**

1. **`e.target` (Целевой узел):** Возвращает строгую ссылку на тот самый DOM-элемент, который физически инициировал событие. Это позволяет использовать одну функцию для обработки кликов по разным элементам.

```javascript
document.body.addEventListener('click', (e) => {
    console.log("Вы кликнули по элементу: ", e.target);
});
```

2. **`e.preventDefault()` (Блокировка дефолтного поведения):**
Многие HTML-элементы имеют встроенное (дефолтное) поведение. Например, клик по ссылке `<a>` вызывает перезагрузку страницы и переход по URL, а нажатие кнопки в `<form>` отправляет данные на сервер с перезагрузкой.
Вызов этого метода немедленно блокирует системное поведение браузера, позволяя вам написать свою собственную логику на JavaScript без перезагрузки интерфейса (что является основой современных SPA-приложений).

```javascript
const form = document.querySelector('.contact-form');
form.addEventListener('submit', (e) => {
    e.preventDefault(); // Страница больше не будет перезагружаться!
    console.log("Данные перехвачены JavaScript'ом");
});
```

### Архитектурный паттерн: Управление состоянием (CSS + JS)

При создании модальных окон, бургер-меню или табов разработчики-новички часто пытаются изменять стили напрямую через JS (например, `modal.style.display = 'block'`). Это считается грубым нарушением архитектуры, так как стили (CSS) смешиваются с логикой (JS).

**Инженерный стандарт (Class-based State Management):**
Правильный подход — заранее описать визуальные состояния (открыто/закрыто) в CSS-файле с помощью классов-модификаторов (например, `.active` или `.is-open`), а с помощью JavaScript только переключать эти классы.

Интерфейс `classList` предоставляет для этого идеальный метод `.toggle()`: он проверяет наличие класса у элемента. Если класса нет — добавляет его, если класс уже есть — удаляет.

**Пример эталонной логики для модального окна:**

```css
/* CSS: Прячем окно по умолчанию */
.modal {
    opacity: 0;
    pointer-events: none;
    transition: opacity 0.3s ease;
}

/* Модификатор активного состояния */
.modal.active {
    opacity: 1;
    pointer-events: auto;
}
```

```javascript
/* JavaScript: Управляем только логикой присутствия класса */
const openBtn = document.querySelector('.btn-open-modal');
const modalBox = document.querySelector('.modal');

// Регистрируем событие клика
openBtn.addEventListener('click', () => {
    // Переключаем класс (Добавит .active при первом клике, уберет при втором)
    modalBox.classList.toggle('active');
});
```
